# 기획서 #7 — 단위/환율 변환 프로그래매틱 SEO (가칭: 바로변환)

> 작성일: 2026-07-10 | 상태: Draft v1 | 배포: Vercel 정적 빌드 (수천 페이지 SSG) + GitHub Actions 환율 갱신, 비용 0원

---

## 1. 문제 정의

- "5피트 cm", "1온스 그램", "100달러 원화", "평수 계산" 같은 변환 검색은 개별 검색량은 작지만 **조합의 수가 무한**하다 (단위쌍 × 수치 = 수만 개 롱테일 키워드).
- 프로그래매틱 SEO의 교과서 시장: 페이지를 템플릿 × 데이터로 자동 생성. unitconverters.net, convertlive 등 영어권 강자는 있으나 **한국어 특화**(평↔㎡, 근·되·홉, 한국 신발 사이즈, 원화 중심 환율)는 품질 낮은 사이트뿐.
- 구글 계산기가 head 키워드(예: "1달러 원")를 가로채지만, 롱테일("3.5평 제곱미터", "원단 1마 cm")과 표·역변환·설명 수요는 여전히 웹사이트로 온다.

## 2. 목표 & 성공 지표

| 지표 | 3개월 | 6개월 |
|------|-------|-------|
| 색인된 페이지 | 2,000 | 10,000+ |
| 월 방문자 | 5,000 | 50,000 |
| 페이지당 평균 체류 | 30초+ | 40초+ |
| 월 AdSense 수익 | 승인 | 30만 원+ |

핵심 리스크 지표: **색인율** (생성 페이지 대비 색인 비율 <30%면 페이지 품질 재설계).

## 3. 타겟 사용자

- 검색 즉답형 사용자: 특정 변환값 하나 필요 (해외직구, 요리, 부동산, 여행)
- 한국 특화 수요: 평수, 전통 단위(근/되), 해외 사이즈표, 원화 환율
- 사용 맥락: 검색 → 착지 → 3초 내 답 확인 → 이탈. 체류시간보다 **즉답 + 관련 변환 유도**가 설계 목표

## 4. 페이지 생성 전략 (프로그래매틱 코어)

### 페이지 타입 3종

**A. 단위쌍 허브** (~500페이지): `/convert/feet-to-cm`
- 인터랙티브 변환기 (클라이언트 JS)
- 변환 공식 + 설명 텍스트 (단위쌍별 고유 문장 — 템플릿 문장 3종 로테이션으로 중복 회피)
- 자주 찾는 수치 변환표 (1, 5, 10, 50, 100...)
- 역변환·관련 단위 내부 링크

**B. 수치 랜딩** (선별 ~5,000페이지): `/convert/feet-to-cm/5`
- "5피트 = 152.4cm" 즉답 H1 + 근처 수치 표
- **선별 기준**: 실검색 수요 있는 수치만 (1–100 정수 + 관용 수치). 무한 생성 금지 — 저품질 대량 색인은 구글 스팸 판정 위험

**C. 한국 특화 스페셜** (~30페이지, 수제): `/평수계산기`, `/신발사이즈표`, `/근-그램`, `/전기요금계산기`
- 경쟁 약한 고가치 키워드. 콘텐츠 품질 최상으로 제작 — 사이트 전체 신뢰도 견인

### 환율 (준실시간, 서버 없이)
- GitHub Actions 1시간 크론 → 한국수출입은행 공개 환율 API(무료) → `rates.json` 커밋 → 재배포
- 페이지에 "기준: YYYY-MM-DD HH:00 매매기준율" 명시 — 실시간 아님을 정직하게 고지

## 5. 기술 스택

| 항목 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Astro SSG | 수천 페이지 정적 생성 최적, 빌드 속도, JS 거의 0 |
| 데이터 | 단위 정의 JSON (팩터·설명·별칭) — 소스 오브 트루스 1파일 | 페이지 = 데이터 × 템플릿 |
| 환율 | 수출입은행 API + Actions 크론 | 무료, 공공 |
| 호스팅 | Vercel (무료 티어: 빌드 45분/100GB 대역폭 — 1만 페이지 정적 OK) | Pages보다 빌드 유연 |
| 사이트맵 | 빌드 시 자동 생성 (분할 sitemap index) | 1만 페이지 색인 필수 |
| 스키마 | `FAQPage`, `HowTo`, breadcrumb | 리치 결과 |

## 6. 정보 구조

```
/                          카테고리 허브 (길이/무게/넓이/온도/환율/사이즈)
/convert/<from>-to-<to>    단위쌍 변환기 (타입 A)
/convert/<from>-to-<to>/<n>  수치 즉답 (타입 B)
/currency/usd-krw          환율 (준실시간 JSON)
/평수계산기 등              한국 특화 (타입 C)
```

- 내부 링크 그물: 수치 페이지 → 허브 → 관련 단위쌍 → 카테고리. 고아 페이지 0 원칙

## 7. 수익화

1. AdSense: 즉답 아래 1슬롯 + 표 아래 1슬롯 (첫 화면 답변 가림 금지 — UX·정책 모두)
2. 맥락 제휴 (쿠팡파트너스): 요리 단위 → 계량컵, 사이즈표 → 직구 관련 상품
3. 환율 페이지 → 트래블카드/환전 제휴 (금융 제휴 단가 높음)

## 8. 경쟁 & 차별화

| 경쟁자 | 상황 | 차별화 |
|--------|------|--------|
| 구글 내장 계산기 | head 키워드 흡수 | 롱테일·표·역변환·한국 특화로 우회 |
| 영어권 대형 (unitconverters.net) | 한국어 미지원/번역 조악 | 네이티브 한국어 설명 + 한국 단위 |

## 12. 핵심 원칙

1. **페이지 = 데이터 × 템플릿**: 단위 정의 JSON이 유일한 소스 오브 트루스. 손으로 페이지 만들지 않는다 (한국 특화 스페셜 제외).
2. **정확도가 생명**: 변환 계수 오류 = 사이트 죽음. Python·JS 양쪽 엔진이 같은 JSON을 읽고, 양쪽 모두 단위 테스트로 교차 검증.
3. **단계 방출**: 전체 생성 후 일괄 배포 금지. 파일럿(수백 페이지) → 색인율 관찰 → 확장. 구글 저품질 판정이 최대 리스크.
4. **즉답 우선**: 수치 페이지는 H1에서 바로 답. 광고·설명이 답보다 위에 오면 실패.
5. **고아 페이지 0**: 모든 생성 페이지는 허브→페어→수치 링크 그물에 연결. 생성기에서 자동 검증.
6. **환율은 정직하게**: 기준 시각·출처 명시, "실시간 아님" 고지. Actions 일일 갱신.
7. **점진 개선**: 단계마다 시각·성능·느낌 +12%, 매 단계 브라우저 실검증.

| 비용 | 0원. 외부 요청은 rates.json(자체 호스팅)뿐 |

## 14. 완성 기준 (Definition of Done)

- [ ] units.json: 카테고리 5(길이·무게·온도·넓이·부피), 단위 30+, 별칭 포함
- [ ] 변환 엔진 2종(Python·JS)이 동일 JSON 사용, 교차 검증 테스트 통과 (온도 등 비선형 포함)
- [ ] generate.py: 페어 허브 + 수치 페이지 + 카테고리 허브 + 사이트맵 자동 생성
- [ ] 파일럿 200~600 페이지, 고아 페이지 0 검증 (생성기 내 링크 그래프 검사)
- [ ] 페어 허브: 인터랙티브 변환기(양방향), 변환표, 공식·설명, FAQPage 스키마, 역변환·관련 단위 링크
- [ ] 수치 페이지: H1 즉답("5피트는 152.4cm"), 근처 수치 표, 허브 역링크
- [ ] 한국 특화 수제 페이지 2+: 평수 계산기(평↔㎡), 환율 페이지(USD/JPY/EUR/CNY→KRW)
- [ ] 환율 파이프라인: Actions 일일 크론, 실패 시 기존 유지, 기준 시각 표기
- [ ] 전 페이지 콘솔 에러 0, 375px 정상, 다크/라이트
- [ ] robots / llms.txt / sitemap(전 페이지 포함 검증)
- [ ] Node + Python 테스트 전부 통과
- [ ] GitHub Pages 배포 + 라이브 검증 (핵심 페이지 200 확인)
